下面这份可以作为你小说平台的 **《阅读 UI 设计调研与体验规范 v1》**。重点不是“做几个设置项”，而是把阅读界面设计成一种长期、低疲劳、可沉浸、可恢复上下文、适配多端的阅读环境。

你的平台定位已经不是普通阅读器，而是“本地上传 + 链接解析 + 免费搜索发现 + 章节净化 + 多端同步 + 沉浸式阅读 + AI 阅读增强”的个人小说聚合阅读平台；已有方案里也把 MVP 聚焦在 TXT/EPUB 上传、自动章节解析、基础阅读器、书架、阅读进度、夜间模式、字体/行距/背景、链接解析、IndexedDB 离线缓存和 AI 章节摘要，这个方向是合理的。

---

# 1. 总体结论：阅读 UI 的核心目标是“让用户忘记 UI”

小说阅读 UI 和普通内容 App 最大的区别是：  
**用户不是来浏览页面的，而是要连续沉浸几十分钟甚至数小时。**

所以阅读 UI 的第一原则不是炫酷，而是：

> **低存在感、低干扰、低疲劳、高可控、强恢复。**

成熟阅读器的共同方向也能印证这一点。Legado/阅读强调高度自定义阅读界面、字体、颜色、背景、行距、段距、简繁转换以及覆盖、仿真、滑动、滚动等多种翻页模式；Koodo Reader 覆盖 Windows、macOS、Linux、Android、iOS 和 Web，并强调同步与备份；Readest 则强调沉浸式深度阅读、批注、笔记、分屏、TTS 和云同步。([github.com](https://github.com/gedoor/legado/blob/master/English.md))

因此，你的阅读 UI 应该分成三层：

| 层级 | 目标 | 设计重点 |
|---|---|---|
| 沉浸阅读层 | 长时间舒服阅读 | 字体、行距、边距、背景、亮度、翻页 |
| 操作控制层 | 需要时快速操作 | 目录、进度、设置、书签、AI、搜索 |
| 内容增强层 | 小说专属体验 | 净化、换源、章节质量、前情提要、人物关系 |

# 2. 阅读视觉：应该是什么感觉？

## 2.1 视觉关键词

你的阅读器视觉不应该像资讯站、论坛、后台系统，而应该接近：

| 关键词 | 说明 |
|---|---|
| 安静 | 少按钮、少边框、少强色块 |
| 柔和 | 背景和文字对比足够但不刺眼 |
| 稳定 | 翻页、滚动、设置变化不突兀 |
| 纸感 | 适度留白、柔和背景、正文居中 |
| 可控 | 用户能精细调整，不被系统强迫 |
| 沉浸 | 阅读时 UI 自动淡出，只保留正文 |
| 可恢复 | 退出、误触、换设备后能回到原位置 |

UI 上要避免“功能很满但阅读很吵”。阅读器首页可以信息丰富，但进入正文后应该尽量克制。

# 3. 正文排版：阅读体验的核心

## 3.1 字号

建议默认值：

| 设备 | 默认字号 | 可调范围 |
|---|---:|---:|
| 手机 | 18px | 14–32px |
| 平板 | 19px | 14–34px |
| Web/桌面 | 20px | 14–36px |
| 老年/大字模式 | 24px | 20–42px |

你已有调研中也把字体大小范围建议为至少 12–36px，并且阅读样式需要支持字体族、行高、段距、页边距、背景色、夜间模式、护眼模式、亮度控制、简繁转换、横竖屏、单栏/双栏等。

我的建议是：**MVP 不要只做 “小 / 中 / 大” 三档字号**。至少要用滑杆或 `- / +` 步进调节，因为小说用户对字号非常敏感。

## 3.2 字体

中文小说建议内置这些字体选项：

| 字体类型 | 适合场景 |
|---|---|
| 系统默认 | 最稳，性能最好 |
| 黑体/苹方/思源黑体 | 现代、清晰、手机阅读 |
| 宋体/思源宋体 | 纸书感、长篇小说 |
| 楷体 | 古风、武侠、仙侠氛围 |
| 自定义字体 | 高级用户需求 |

不要默认使用过于花哨的字体。小说正文最重要的是 **字形稳定、笔画清晰、长时间不累**。

## 3.3 行高

建议默认：

| 场景 | 行高 |
|---|---:|
| 手机小说 | 1.65 |
| 平板 | 1.6 |
| 桌面单栏 | 1.7 |
| 桌面双栏 | 1.55 |
| 大字模式 | 1.75 |

Material Design 3 和 Apple HIG 都强调排版层级、行高和可读性；Apple 明确建议三行以上文本避免过紧的 leading，WCAG 则强调文本与背景需要足够对比，以便低视力和对比感知较弱的用户阅读。([m3.material.io](https://m3.material.io/styles/typography/applying-type))

我的建议是：**行高默认 1.6–1.75，允许用户调到 1.2–2.4**。你已有方案里也把行高范围列为 1.2–2.4，这是合理的。

## 3.4 段距

中文网文常见问题是段落很短、换行频繁，如果段距过大，会导致阅读节奏碎片化。

建议：

| 模式 | 段距 |
|---|---:|
| 紧凑 | 0.2em |
| 默认 | 0.5em |
| 舒适 | 0.8em |
| 纸书 | 1em |
| 网文原排版 | 保留原始空行 |

特别要支持“空行压缩”：在线小说经常有大量空行、广告空白、分页残留，UI 层不应完全相信原文排版。

## 3.5 行宽

行宽是桌面端和横屏端最容易被忽略的点。

建议：

| 场景 | 正文宽度 |
|---|---:|
| 手机竖屏 | 占屏幕宽度 100%，左右 18–24px 边距 |
| 手机横屏 | 最大 720px，居中 |
| 平板竖屏 | 最大 760px，居中 |
| 平板横屏 | 单栏 760px 或双栏 |
| 桌面单栏 | 680–820px |
| 桌面双栏 | 每栏 360–460px |

可读性研究和设计实践通常建议正文行长不要过长，Baymard 总结正文理想行长大约 50–60 字符，其他来源也常将 45–75 字符作为舒适区间。([baymard.com](https://baymard.com/blog/line-length-readability))

中文不是按英文字符完全等价，但原则一样：**桌面端绝不能让正文铺满 1440px 宽屏**。否则用户视线移动过长，读几章就会疲劳。

# 4. 色彩与主题：不要只做浅色/深色

## 4.1 推荐主题

至少提供这些预设：

| 主题 | 背景 | 文字 | 适合场景 |
|---|---|---|---|
| 纸白 | #F8F8F5 | 深灰 | 白天、桌面 |
| 米黄 | #F4ECD8 | 棕黑 | 长篇小说、纸书感 |
| 护眼绿 | #DDEBD6 | 深灰绿 | 部分用户偏好 |
| 暖灰 | #E8E3DA | 深灰 | 夜晚低刺激 |
| 深灰夜间 | #1E1E1E | #CFCFCF | OLED/夜间 |
| 纯黑夜间 | #000000 | #BDBDBD | OLED 省电 |
| 自定义 | 用户配置 | 用户配置 | 高级用户 |

注意：夜间模式不要用纯白文字配纯黑背景作为默认，因为对比过强会刺眼。更好的默认是深灰背景 + 柔和浅灰文字。

WCAG 2.2 对普通文本的最低对比度要求是 4.5:1，大号文本为 3:1；这应该作为主题校验的底线，而不是视觉建议的上限。([w3.org](https://www.w3.org/TR/WCAG22/))

## 4.2 亮度遮罩

移动端阅读器很需要 App 内亮度控制，尤其是 PWA/浏览器场景。

建议实现：

```ts id="06vloq"
ReaderBrightness = {
  mode: 'system' | 'overlay',
  overlayOpacity: 0.0 - 0.65,
  warmth: 0.0 - 1.0
}
```

在 Web/PWA 里不能直接控制系统亮度，可以用半透明遮罩模拟。这个功能对夜间阅读非常实用。

# 5. 阅读主界面布局

## 5.1 默认沉浸态

阅读页面默认只显示正文。

可选弱显示：

| 元素 | 显示方式 |
|---|---|
| 当前章节名 | 顶部小字，默认可关闭 |
| 当前页/进度 | 底部极弱显示 |
| 时间/电量 | 移动端可选 |
| 下一章提示 | 仅章末出现 |
| AI/设置按钮 | 不常驻，呼出菜单后显示 |

阅读中不要常驻大按钮、浮动工具栏、广告、推荐卡片、强提醒。

## 5.2 呼出菜单态

移动端建议点击屏幕中央呼出菜单。你已有调研建议的点击区域是：左 25% 上一页/上一屏，中 50% 呼出菜单，右 25% 下一页/下一屏；底部菜单放目录、进度、设置、夜间、书签、更多；顶部菜单放返回、书名、搜索、AI、更多。这个交互可以直接作为 MVP 方案。

推荐结构：

```text id="vcvm2k"
顶部栏：
返回 / 书名 / 搜索 / AI / 更多

正文区域：
左侧点击区：上一页
中间点击区：菜单
右侧点击区：下一页

底部栏：
目录 / 进度 / 设置 / 夜间 / 书签 / 更多
```

## 5.3 桌面端布局

桌面端不要简单放大移动端。建议三种模式：

| 模式 | 适用 |
|---|---|
| 沉浸单栏 | 默认阅读 |
| 左目录 + 中正文 | 查找章节、长篇跳转 |
| 中正文 + 右 AI/笔记 | 摘要、人物、设定、批注 |
| 双栏阅读 | 宽屏、平板横屏、仿纸书 |
| 全屏阅读 | 长时间沉浸 |

你已有调研也建议桌面端支持单栏居中、双栏阅读、左目录 + 右正文、右侧笔记/AI 面板和沉浸全屏。

# 6. 翻页与滚动设计

## 6.1 必须支持的阅读模式

| 模式 | 优先级 | 说明 |
|---|---:|---|
| 上下滚动 | P0 | Web/PWA 最自然 |
| 左右分页 | P0 | 小说 App 常用 |
| 点击翻页 | P0 | 单手阅读关键 |
| 滑动翻页 | P0 | 移动端自然交互 |
| 键盘翻页 | P0 | 桌面端必须 |
| 自动滚动 | P1 | 吃饭、通勤、躺读 |
| 音量键翻页 | P1 | Android 重点 |
| 仿真翻页 | P2 | 情绪价值高，但非核心 |

Legado 支持覆盖、仿真、滑动、滚动等多种翻页模式；Koodo/KOReader/Readest 等阅读器也都强调不同设备上的排版、翻页和沉浸阅读能力。([github.com](https://github.com/gedoor/legado/blob/master/English.md))

我的建议：  
**MVP 先做好滚动 + 分页，不要一开始投入太多在仿真翻页。** 仿真翻页视觉成本高、分页计算复杂、性能风险大，但对核心阅读留存不如字号/行距/进度稳定重要。

## 6.2 分页模式关键细节

分页不是简单 `overflow: hidden`。

需要注意：

| 问题 | 设计要求 |
|---|---|
| 字号改变 | 重新分页并尽量保持阅读位置 |
| 横竖屏切换 | 重新计算页数 |
| 图片/插图 | 不应被截断得太难看 |
| 章节末尾 | 显示下一章入口 |
| 跨页高亮 | 后期要支持 |
| 进度保存 | 保存到段落/字符偏移，而不是只存页码 |

Koodo 的用户 issue 中有人提到 EPUB 在双页模式下跨页高亮的问题，这类细节说明深度阅读产品后期一定会遇到“分页与选区/批注”的复杂交互。([github.com](https://github.com/koodo-reader/koodo-reader/issues/1458))

# 7. 目录、进度与导航

小说用户经常读几百章、上千章，导航体验必须很强。

## 7.1 目录 UI

目录不要只是章节列表，应包含：

| 能力 | 说明 |
|---|---|
| 当前章自动定位 | 打开目录滚到当前章 |
| 章节搜索 | 搜“番外”“大结局”“第 300 章” |
| 分卷折叠 | 长篇小说必要 |
| 已读/未读状态 | 视觉区分 |
| 异常章节标识 | 缺章、重复、过短、广告 |
| 来源标识 | 当前章节来自哪个源 |
| 手动修正入口 | 标题错、边界错时可改 |

## 7.2 进度条

建议做两级进度：

```text id="26u6j6"
全书进度：第 583 / 1600 章，36.4%
当前章进度：本章 72%
```

底部进度条可以增加章节刻度，但不要太花。长篇小说的章节数量巨大，刻度需要抽样或按卷聚合。

## 7.3 回退机制

这是很重要但常被忽略的体验。

用户从目录跳到第 1000 章、又点了搜索结果、又打开 AI 面板后，应该能“一键回到刚才阅读位置”。

建议内部维护：

```ts id="aebkal"
type ReadingLocationStackItem = {
  bookId: string
  chapterId: string
  offset: number
  reason: 'toc' | 'search' | 'bookmark' | 'ai' | 'manual'
  createdAt: number
}
```

# 8. 设置面板设计

## 8.1 设置不要做成复杂表单

阅读设置应该是“边调边看”。

建议底部弹出半屏面板：

```text id="ra9x9s"
第一层：常用
字号 - / +
行距 - / +
背景主题
夜间模式
翻页方式

第二层：高级
字体
段距
页边距
正文宽度
段首缩进
简繁转换
亮度遮罩
保留原排版
强制覆盖 EPUB 样式
```

KOReader 用户指南里提到可以为字体、行距、对比度、粗细等创建配置文件，并放进快捷菜单；这说明高级阅读器用户会需要“阅读配置档”，而不是每本书重复调。([koreader.rocks](https://koreader.rocks/user_guide/))

## 8.2 阅读配置档

建议支持：

| 配置档 | 用途 |
|---|---|
| 默认 | 普通小说 |
| 夜间 | 深色 + 大行距 |
| 纸书 | 宋体 + 米黄 + 较大边距 |
| 紧凑 | 小字号 + 小段距 |
| 大字 | 大字号 + 大行距 |
| 自定义 1/2/3 | 高级用户 |

还要支持：

```text id="ekhs9j"
应用到当前书
应用到所有书
仅当前设备
同步到所有设备
```

用户会非常介意“我调好的样式为什么下本书不生效”。KOReader 社区也有用户讨论默认字体、字号、行距、边距在不同书籍中表现不一致的问题。([reddit.com](https://www.reddit.com/r/koreader/comments/1gpsvzo/default_document_settings/))

# 9. 书架 UI 设计

阅读器不只正文页，书架也是关键入口。

## 9.1 书架首页

推荐布局：

```text id="kz77nb"
顶部：
搜索 / 导入 / 设置

继续阅读：
最近一本书大卡片，显示当前章节、进度、上次阅读时间

书架：
网格 / 列表切换
分类筛选：全部、追更、待读、已完结、本地、在线
排序：最近阅读、最近更新、导入时间、书名、作者
```

## 9.2 书籍卡片

卡片信息建议：

| 信息 | 必要性 |
|---|---|
| 封面 | P0 |
| 书名 | P0 |
| 作者 | P1 |
| 当前章 | P0 |
| 阅读进度 | P0 |
| 更新状态 | P1 |
| 来源类型 | P0 |
| 异常提示 | P1 |
| 本地/在线/缓存状态 | P1 |

小说平台区别于普通阅读器的一点是：在线来源可能失效、章节可能坏、更新可能延迟。所以书架卡片上可以出现弱提示：

# 10. 链接解析与在线阅读 UI

你的平台支持粘贴小说链接，这块 UI 要特别注意“可解释”。

## 10.1 解析过程 UI

不要只显示 loading。建议分步骤：

```text id="sfg2n5"
正在识别页面类型
正在提取书名/作者
正在查找目录
正在解析章节列表
正在清理正文
正在评估章节质量
```

解析失败时要告诉用户原因：

| 失败原因 | 用户提示 |
|---|---|
| CORS 限制 | 需要后端代理解析 |
| 动态渲染 | 页面内容由 JS 加载，需服务端渲染兜底 |
| 没找到目录 | 可选择“只读当前页”或手动选择目录 |
| 正文过短 | 疑似广告页/跳转页 |
| 反爬 | 当前来源暂不可用，可换源 |

你已有调研中已经指出，链接解析不能只靠通用正文提取，因为小说需要书名、作者、目录、章节标题、正文、上一章/下一章、分页正文、章节顺序等结构化信息，并建议采用“书源规则优先 -> 站点适配器 -> 通用启发式 -> Readability 兜底 -> 内容净化 -> 质量评分”的流程。

## 10.2 解析结果预览

加入书架前应展示：

```text id="0qzgkn"
书名
作者
封面
简介
目录数量
首章预览
最近章节
来源站点
质量评分
是否可持续更新
```

这能避免用户加了一本“目录错乱/正文不干净”的书。

# 11. 小说专属 UI：章节质量、净化、换源

这是你和普通 EPUB 阅读器拉开差距的地方。

## 11.1 章节质量提示

不要把技术检测藏在后台。应该用轻量 UI 告诉用户：

| 异常 | UI 表现 |
|---|---|
| 缺章 | 目录中显示缺失标记 |
| 重复章 | 章节标题旁提示“疑似重复” |
| 断章 | 章末提示“疑似未完” |
| 广告污染 | 提供“一键净化” |
| 乱码 | 提供“重新识别编码” |
| 错章 | 提供“换源/报错/手动修正” |

你已有调研也明确指出，中文网文常见缺章、错章、重复章、广告章，章节质量检测、换源、净化是小说平台区别于普通电子书阅读器的关键。

## 11.2 净化 UI

净化不要只有一个开关，建议：

```text id="stygl7"
净化模式：
关闭
轻度：去广告节点、空行压缩
标准：去水印、合并分页、修正段落
强力：删除疑似广告段、重复段
自定义：用户规则
```

Legado 的能力里包括替换净化、简繁转换和高度自定义阅读界面，这对中文小说阅读器是很重要的 benchmark。([github.com](https://github.com/gedoor/legado/blob/master/English.md))

## 11.3 换源 UI

换源界面建议按“章节质量”排序，而不是只按来源站点列出。

```text id="58837k"
当前章节可用来源：
来源 A：完整，广告少，更新快
来源 B：完整，但广告较多
来源 C：疑似缺字
来源 D：加载失败
```

后期可以支持：

```text id="25knyl"
仅当前章换源
从当前章起使用此源
整本书切换到此源
缺章时自动补源
```

# 12. AI 阅读 UI：不要做成一个孤立聊天框

你的已有方案里已经提到 AI 不要做噱头，而要嵌入阅读体验：摘要、前情、人物、去水、推荐、创作；并且“去水版/精简版”不应覆盖原文，而应作为衍生阅读视图。

## 12.1 AI 入口

建议放在：

| 位置 | 功能 |
|---|---|
| 顶部栏 AI | 打开 AI 面板 |
| 章首卡片 | 前情提要 |
| 章末卡片 | 本章摘要 |
| 选中文本菜单 | 解释、翻译、加入设定 |
| 右侧面板 | 人物、设定、伏笔、时间线 |

## 12.2 AI 面板结构

不要默认遮挡正文。桌面端用右侧栏，移动端用底部抽屉。

```text id="sbliuk"
AI 面板：
前情提要
本章摘要
人物关系
设定/术语
伏笔
时间线
剧情速读
去水版
```

## 12.3 AI 视觉原则

AI 生成内容要和原文明确区分：

| 内容 | 视觉 |
|---|---|
| 原文 | 正文样式 |
| AI 摘要 | 卡片/侧栏 |
| 去水版 | 顶部显示“AI 衍生视图” |
| AI 修改内容 | 不覆盖原文 |
| 不确定内容 | 明确标识“可能不完整” |

# 15. 用户反馈与社区意见归纳

综合成熟阅读器和用户讨论，真实用户最在意的不是某个单点功能，而是以下几类：

| 用户诉求 | 对 UI 的要求 |

| 字体字号舒服 | 细粒度调节、配置档 |

| 夜间不刺眼 | 深灰夜间、亮度遮罩 |

| 目录准确 | 当前章定位、搜索、异常提示 |
| 在线章节干净 | 净化、换源、质量评分 |
| 长篇不迷路 | 前情提要、人物、伏笔、时间线 |
| 无广告干扰 | 阅读页保持纯净 |
| 私有/自用 | 数据可导出、无强制账号 |

你的已有调研也归纳过：阅读设置要足够细，同步必须可靠，免费/无广告/无需注册会提升好感，用户要导入自由，在线小说用户需要章节质量保障。

# 18. 最终建议：你的阅读 UI 应该这样定型

我建议你的阅读器设计方向定为：

> **默认极简沉浸，设置高度可调，导航强大但隐藏，小说质量问题可视化，AI 只在需要时作为阅读增强出现。**

具体落地原则：

第一，正文页要克制。阅读中不要常驻复杂工具栏，不要推荐流，不要广告，不要强运营位。

第二，排版设置要专业。字号、行高、段距、边距、字体、背景、亮度、宽度都要可调，并支持配置档。

第三，移动端优先单手阅读。左/中/右点击区、底部菜单、半屏设置面板是最稳的方案。

第四，桌面端不要铺满宽屏。必须有正文最大宽度、单栏居中、双栏、左目录、右 AI/笔记面板。

第五，小说平台必须显示章节质量。缺章、错章、重复、广告、断章、乱码都应该有 UI 提示和修复入口。

第六，AI 不要做成聊天机器人。AI 应该是“前情提要、本章摘要、人物关系、设定、伏笔、去水版”等嵌入式阅读工具。

第七，MVP 先把“持续读得舒服”做到极致。仿真翻页、复杂动画、主题市场、社交功能都可以后置；TXT/EPUB 解析、目录、阅读进度、排版、离线缓存、夜间模式必须先稳。
