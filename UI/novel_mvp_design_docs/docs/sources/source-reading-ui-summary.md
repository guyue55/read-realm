是的，整体设计就是这个思路：

> **前端优先处理，后端兜底增强；前端产生的有效结果再同步到后端，形成跨端可恢复、可复用的数据资产。**

不过要拆细一点，不是所有能力都适合前端先做。合理分工应该是下面这样。

## 1. 前端优先处理什么

前端适合处理 **本地、即时、隐私敏感、用户交互强** 的任务。

| 能力 | 前端职责 |
|---|---|
| TXT / EPUB 上传解析 | 读取文件、识别编码、解析目录、切章、生成预览 |
| 阅读器渲染 | 滚动/分页、字号、行高、边距、主题、夜间模式 |
| 本地离线缓存 | IndexedDB / OPFS 保存书籍、章节、进度、设置 |
| 阅读进度 | 高频保存当前章、偏移、百分比、锚点 |
| 书签/笔记 | 本地即时创建，稍后同步 |
| 章节净化初步处理 | 空行压缩、广告文本规则、HTML 清理 |
| URL 解析尝试 | 对允许跨域的页面尝试前端解析 |
| 解析预览与手动修正 | 用户确认章节、标题、正文质量 |

之前方案里已经明确建议：用户上传 TXT / EPUB / HTML / MD 应优先前端处理，因为文件已在本地，隐私好、速度快；而粘贴网页链接则前端尝试，后端代理兜底。

## 2. 后端兜底处理什么

后端适合处理 **跨域、长任务、重计算、需要统一管控、需要持久同步** 的任务。

| 能力 | 后端职责 |
|---|---|
| URL 代理解析 | 解决 CORS、编码、动态渲染、反爬、站点适配 |
| 搜索聚合 | 搜索引擎/免费接口/书源规则统一管理，避免暴露 key |
| 章节批量抓取 | 队列、并发、重试、缓存、失败恢复 |
| PDF / DOCX / MOBI / AZW3 转换 | Calibre / 服务端转换 / 沙箱处理 |
| AI 摘要与前情 | 模型调用、成本控制、缓存、权限控制 |
| 账号同步 | 书架、进度、设置、书签、笔记、AIView 同步 |
| 多端数据恢复 | 新设备登录后恢复书架和阅读进度 |
| 质量检测增强 | 缺章、重复、乱码、广告章、断章检测 |
| 安全与合规 | 来源限制、用户数据隔离、任务审计 |

PRD 里也已经把前后端分工定为：TXT/EPUB 上传解析前端优先，URL 正文提取前端尝试、后端兜底，PDF/DOCX/MOBI/AZW3 部分前端尝试但主要后端转换，AI 摘要和搜索聚合后端优先。

## 3. 前端结果要不要同步到后端

要，但不是所有内容都无脑同步。建议分为三类。

### A. 必须同步

这些是跨端体验的核心：

| 数据 | 原因 |
|---|---|
| 书架元数据 | 多端恢复书架 |
| 阅读进度 | 手机/Web/PWA 续读 |
| 阅读设置 | 用户排版偏好可复用 |
| 书签/笔记 | 用户资产，不能只存在本地 |
| 章节目录结构 | 避免每个设备重复解析 |
| 手动修正结果 | 用户改过的章节标题/边界应成为可信版本 |
| AI 摘要缓存 | 避免重复生成和重复消耗 |

调研里也明确了云端同步优先级：书架和阅读进度是 P0，阅读设置、书签/笔记是 P1，AI 摘要/人物关系是 P2。

### B. 视情况同步

| 数据 | 建议 |
|---|---|
| 上传原文件 | 看用户设置、存储成本和隐私策略 |
| 全量章节内容 | 默认不一定同步，用户开启云书库后再同步 |
| URL 抓取章节缓存 | 可同步目录和来源，不一定同步全部正文 |
| 解析中间日志 | 只保留错误摘要或质量评分，不保留敏感全文 |
| 封面图片 | 可同步或对象存储缓存 |

### C. 默认只保留本地

| 数据 | 原因 |
|---|---|
| 高频阅读滚动位置临时状态 | 写太频繁，不适合每次上云 |
| 分页后的虚拟页码 | 受设备、字号、行宽影响，不可靠 |
| 本地渲染缓存 | 与设备相关，后端没必要保存 |
| 临时 URL 预览内容 | 用户未确认加入书架前，不应默认入库 |

特别注意：**页码不应该作为真实同步进度**。应该同步 `bookId + chapterId + offset/CFI/textAnchor + percentage + updatedAt`，因为字体、行高、边距、单双栏和屏幕尺寸变化都会导致页码漂移。你前面的阅读功能调研里已经指出，页码只适合作为当前设备展示层，真实进度应使用 CFI、章节位置、文本锚点和百分比。

## 4. 推荐的数据流

比较稳的流程是：

```text id="hsqrph"
前端处理
  -> 本地保存 IndexedDB / OPFS
  -> 生成标准化结果
  -> 用户确认
  -> 后台同步到后端
  -> 后端校验 / 补全 / 入库
  -> 其他设备可恢复
```

例如 TXT 上传：

```text id="q3eflu"
用户上传 TXT
  -> 前端识别编码
  -> 前端切章
  -> 前端净化
  -> 前端生成 Book + Chapter[]
  -> 用户在解析预览页确认
  -> 本地 IndexedDB 保存
  -> 后端同步 Book 元数据、TOC、章节 hash、用户修正
  -> 如果用户开启云书库，再上传原文件或章节内容
```

例如 URL 解析：

```text id="mc8uch"
用户粘贴 URL
  -> 前端尝试 fetch / Readability / 规则解析
  -> 成功则展示预览
  -> 失败则调用后端代理
  -> 后端处理 CORS / 动态渲染 / 编码 / 反爬
  -> 返回标准化 Book / Chapter / Toc
  -> 前端确认加入书架
  -> 后端保存来源、目录、质量评分、缓存策略
```

## 5. 同步策略建议

不要做“实时全量同步”，而应做 **本地优先 + 增量同步 + 冲突解决**。

建议同步对象都带这些字段：

```ts id="r82lqn"
type SyncableEntity = {
  id: string
  userId: string
  version: number
  updatedAt: string
  deletedAt?: string
  deviceId: string
  dirty?: boolean
}
```

同步策略：

| 场景 | 策略 |
|---|---|
| 阅读进度 | 本地节流保存，每 5–15 秒或章节切换时同步 |
| 设置变更 | 立即本地生效，短延迟同步 |
| 书签/笔记 | 创建后本地立即显示，后台同步 |
| 章节解析结果 | 用户确认后同步 |
| 手动修正 | 高优先级同步，覆盖自动解析结果 |
| AI 摘要 | 后端生成后直接入库，前端缓存副本 |
| 冲突 | 进度按 `updatedAt` 或用户选择；笔记保留双份；章节修正用版本号 |

## 6. 最终架构原则

可以把系统定成这四句话：

1. **前端是第一阅读现场**：负责导入、解析尝试、阅读渲染、离线缓存、即时交互。
2. **后端是可靠性和跨端层**：负责代理、转换、搜索、AI、同步、队列、缓存和兜底。
3. **前端结果必须标准化后再同步**：不要把页面临时状态直接同步，要同步 Book、Chapter、Progress、Bookmark、Annotation、AIView 这些稳定结构。
4. **同步以用户确认和低频关键状态为主**：进度、书架、设置、笔记、目录、修正结果优先；全量正文和原文件可选。

所以答案是：**是的，前端会承担相当一部分功能；前端搞不定时交给后端；前端处理出来的有效结构化结果也会同步到后端，但需要分级、节流、确认和冲突处理。**
