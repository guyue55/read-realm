# 安全策略 (Security Policy)

我们非常重视 **我的阅读世界 (read-realm)** 的安全。如果由于代码缺陷导致系统安全受损，或者出现可被利用的漏洞，我们希望能够尽快查明并予以修复。

---

## 1. 负责任的披露流程 (Responsible Disclosure)

**请勿直接在 GitHub Issue 中公开提交安全漏洞。** 公开披露漏洞可能会使所有正在使用本项目的用户面临风险。

如果你发现了任何潜在的安全问题：

1. 请发送私密邮件至项目维护者邮箱：`guyue_admin@example.com`（请根据实际维护者邮箱替换）。
2. 邮件中请包含：
   - 漏洞的详细描述
   - 漏洞影响的分支、应用端（如 Web-PWA, API 等）
   - 尽可能详细的复现步骤 (Proof of Concept, PoC)
   - 任何可能的修复建议
3. 我们会在收到邮件后的 **48 小时内** 给予答复，并与你协作跟进修复。
4. 在漏洞修复并发布安全更新后，我们会与你共同决定是否进行公开披露，并在 Release Notes 中对你致谢。

---

## 2. 关注的核心安全领域

在进行开发或使用时，我们尤其关注以下安全模块：

- **净化/规则引擎沙箱**：由于书源与智能净化规则（在 `packages/rules-engine`）可能执行复杂的解析，必须保证其在一个安全的、隔离的沙箱中运行，严防 JavaScript 沙箱逃逸或远程恶意代码执行（RCE）。
- **文件解析安全性**：在 `packages/parser-core` 中解析 TXT、EPUB 文件时，应防止 XML 外部实体注入（XXE）、Zip Slip（Zip 路径穿透）等传统文件解析器漏洞。
- **离线存储与 XSS 净化**：在客户端（`apps/web-pwa`）存储、展示小说正文前，必须通过严格的 HTML 净化（如 DOMPurify 等），防止恶意 HTML/JS 注入 IndexedDB/OPFS。
- **API SQL 注入**：NestJS 后端（`apps/api`）由于使用 SQLite + Drizzle ORM，应确保所有的 FTS5 全文检索和数据库查询都通过参数化查询，防止 SQL 注入。

## 3. 受支持的版本

目前，仅以下版本接受积极的安全维护：

| 版本                    | 是否支持安全修复 | 备注                                                 |
| :---------------------- | :--------------: | :--------------------------------------------------- |
| `0.1.x` (当前 MVP 分支) |       Yes        | 处于快速开发验证期，任何漏洞都会作为最高优先级修复。 |
| `< 0.1.0`               |        No        | 历史实验版本，请尽快升级。                           |
